## Part 1 — What is Version Control?

Before touching any tools, it is important to understand *why* they exist.

Imagine you are building a website with three teammates. You are each editing HTML, CSS, and JavaScript files. Without any system in place, you would be emailing files back and forth, overwriting each other's work, and losing track of who changed what — and when. That is exactly the chaos version control was invented to prevent.

**Version control** is a system that records changes to files over time so you can:

- Recall any earlier version of your project at any point
- See who made every change and when
- Collaborate with teammates without overwriting each other's work
- Experiment freely without breaking the stable version of your code

---

## Part 2 — What is Git?

**Git** is the world's most widely used version control system. It was created in 2005 by Linus Torvalds — the same person who created Linux — to manage the development of the Linux kernel.

Git runs **locally on your machine**. It watches your project folder and records a snapshot every time you tell it to. These snapshots are called **commits**.

!!! tip "Git vs. GitHub — Know the Difference"
    - **Git** is the version control tool that runs on your computer.
    - **GitHub** is a cloud platform that hosts your Git repositories online.

    Think of Git as the engine, and GitHub as the garage where you park — and share — your cars.

---

## Part 3 — What is GitHub?

**GitHub** is a web-based platform built on top of Git. It stores your code in the cloud and adds powerful features for collaboration.

Here is why GitHub matters to you as a developer:

| Feature | What it means for you |
|---|---|
| ☁️ **Cloud Backup** | Your code is safe even if your laptop dies |
| 👥 **Collaboration** | Multiple people can work on the same project |
| 📜 **Full History** | See every change ever made, and who made it |
| 🔀 **Branching** | Experiment freely without breaking the main code |
| 🌍 **Portfolio** | Employers look at GitHub profiles — yours will be ready |

GitHub is used by millions of developers and companies worldwide — from solo students to Netflix, Google, and Microsoft. Learning it now is not optional; it is an industry-standard skill.

## Part 4 — Core Concepts

You will see these terms constantly throughout this course. Learn them now.

**Repository (Repo)**
:   A repository is a project folder tracked by Git. It contains all your files and the complete history of every change ever made to them.

**Commit**
:   A commit is a saved snapshot of your project at a specific moment. Every commit has a message describing what changed. Think of it like pressing Save — but with a description, a timestamp, and an author attached.

**Push**
:   Pushing means sending your local commits up to GitHub so others can see your work.

**Pull**
:   Pulling means downloading the latest changes from GitHub to your local machine so you stay up to date.

**Clone**
:   Cloning means downloading a full copy of a GitHub repository to your local machine, with Git tracking already built in.

## Part 5 — Setting Up Git on Your Machine

### Step 1 — Install Git

=== "Windows"
    1. Go to [https://git-scm.com/download/win](https://git-scm.com/download/win) and download the installer
    2. Run the installer — the default options are fine
    3. When done, open **Git Bash** from your Start menu and use it as your terminal for all commands below

=== "macOS"
    Open **Terminal** and run:
    ```bash
    xcode-select --install
    ```
    Git is included with Xcode Command Line Tools. If it is already installed, you will see a message confirming it.

    You can find more information on how to install Git on macOS at [https://git-scm.com/](https://git-scm.com/).

=== "Linux (Ubuntu / Debian)"
    ```bash
    sudo apt update
    sudo apt install git
    ```

### Step 2 — Verify Installation

Open your terminal and run:

```bash
git --version
```

You should see something like `git version 2.43.0`. If you do, Git is installed successfully.

### Step 3 — Configure Git with Your Identity

Git needs to know who you are so your commits are correctly attributed. Run these two commands, replacing the values with your own name and email:

```bash
git config --global user.name "Your Full Name"
git config --global user.email "your@email.com"
```

!!! warning "Use the Same Email as GitHub"
    Use the same email address you will register with on GitHub. This links your commits to your GitHub profile correctly.

Confirm your configuration was saved:

```bash
git config --list
```

You should see your name and email listed in the output.

## Part 8 — Cloning the Repository to Your Machine

Cloning downloads a copy of your GitHub repository to your local computer, with Git tracking already set up.

**Step 1 — Copy the Repository URL**

On your repository page, click the green **Code** button and copy the HTTPS URL. It will look like:

```
https://github.com/your-username/web-dev-tutorial-01.git
```

**Step 2 — Clone it Locally**

Open your terminal, navigate to a folder where you keep your coursework, and run:

```bash
cd ~/Documents/webdev        # navigate to your coursework folder
git clone https://github.com/your-username/web-dev-tutorial-01.git
cd web-dev-tutorial-01
```

**Step 3 — Confirm it Worked**

```bash
ls        # macOS / Linux
dir       # Windows
```

You should see your `README.md` file listed. You are now inside your local repository.

## Part 9 — Adding Files and Making a Commit

Now you will create a file, commit it, and push it to GitHub.
### Step 0 — Edit the README.md file

Open `README.md` in your code editor and add the following content **exactly as written** (replace the bracketed text with your own information):

```markdown
# My First GitHub Page

## My name is [Your Name] - [your Algoma email]

### COSC2956 - Assignment 4

This file was committed and pushed from my local machine.
```


### Step 1 — Create a New File

Inside your repository folder, create a file called `index.html`:

Open `index.html` in your code editor and add the following content **exactly as written** (replace the bracketed text with your own information):

```html
<!DOCTYPE html>
<html lang="en">
<head>
    <meta charset="UTF-8" />
    <title>My First GitHub Page</title>
</head>
<body>
    <h1>Hello, GitHub!</h1>
    <h2> My name is [Your Name] - [your Algoma email]</h2>
    <h3>COSC2956 - Assignment 4</h3>
    <p>This file was committed and pushed from my local machine.</p>
</body>
</html>
```

Save the file.

### Step 2 — Check the Status

```bash
git status
```

Git will show `index.html` as an **untracked file** — it exists in the folder but Git is not watching it yet.

### Step 3 — Stage the File


To stage all changed files at once:

```bash
git add .
```

### Step 4 — Commit the Change

```bash
git commit -m "Add index.html with basic HTML structure"
```

!!! tip "Write Meaningful Commit Messages"
    Your commit message should complete the sentence: *"This commit will..."*

    ✅ `"Add index.html with basic HTML structure"`

    ❌ `"stuff"` &nbsp; ❌ `"fix"` &nbsp; ❌ `"asdfjkl"`

### Step 5 — Push to GitHub

```bash
git push origin main
```

Now visit your repository on GitHub and refresh the page. You will see `index.html` has appeared.

## Summary — The Core Git Workflow

Every time you work on a project, your workflow follows this loop:

```
1.  Make changes to your files
          ↓
2.  git add .               ← Stage your changes
          ↓
3.  git commit -m "..."     ← Save a snapshot with a message
          ↓
4.  git push origin main    ← Upload to GitHub
```

And when you need to get the latest changes from GitHub:

```bash
git pull origin main        # Download the latest from GitHub
```

This loop is the heartbeat of modern software development. Every project in this course from here on will use it.
